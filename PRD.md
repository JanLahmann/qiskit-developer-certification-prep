# PRD — Qiskit v2.x Certification Prep Platform

> Working title: **Qiskit Certification Studio** (final name chosen at v1 announcement — candidates in §9).
> Status: living document, created 2026-07-19. Origin: doQumentation `BUILD_TIME_AI_IDEAS.md` §2, built here as a standalone project.
> **Unofficial community project — not affiliated with, endorsed by, or sponsored by IBM.**

## 1. Mission & thesis

A free, high-end, **guided** learning platform for the IBM Qiskit v2.x Developer certification (exam C1000-179) — not another list of sample questions.

The engine is **build-time AI, zero runtime AI**: every intelligent behavior (question generation, answer proofs, remediation advice, study sequencing, difficulty calibration) is precomputed by frontier-model agents during the build and shipped as static data. The site is pure static Docusaurus + client-side JS; personalization happens by *selection* from precomputed decision tables, keyed off localStorage. No accounts, no backend, no tracking, works offline.

**The moat: machine-verified truth.** Every code-bearing question is executed against a pinned Qiskit 2.x stack at build time — the correct answer is *proven*, the distractors are *disproven*, and the proof ships with the question. Every commercial question bank silently rots; ours is freshness-gated by CI on every Qiskit release.

## 2. Users

1. **Cert candidate** — wants a structured path from "registered" to "passed", trustable practice, and a realistic readiness signal.
2. **Aspiring Qiskit Advocate** — needs the cert to reach Tier 1; also needs contribution opportunities (our community-exam validation reports are PR fodder).
3. **Study-group organizer** — needs per-section material to assign, and per-member progress export.

## 3. Non-negotiables

- **NDA-clean**: questions generated ONLY from public exam objectives + official docs + our own expertise. Never from real exam content. Say so prominently.
- **Unofficial disclaimer** on every page footer + legal page (wording pattern adopted from doQumentation): *"This is an independent community project. It is not affiliated with, endorsed by, or sponsored by IBM. Qiskit is a trademark of IBM."*
- **Static-only runtime**: GitHub Pages, no server, no runtime AI, no accounts. All state in namespaced localStorage (`qcs.*`).
- **Provenance labels**: content marked "AI-generated at build time, execution-verified" (and "human-reviewed" once true). Proof metadata visible per question.
- **License hygiene**: our code Apache-2.0; our original content CC BY-SA 4.0. Official docs: deep-link, don't excerpt prose (CC BY-SA share-alike); short Apache-2.0 code snippets OK with attribution. Community exams: link + credit, never republish their questions (most have no license).
- **Accessibility**: keyboard-first quiz/exam player, alt text, sensible contrast in both themes.

## 4. Features

### F1 — Syllabus Explorer & guided path (the spine)
- One page per exam section (8 sections from `data/syllabus.json`): official objectives, weight, what-to-master scope notes, curated official resources (docs / IBM Quantum Learning / videos), then practice sets.
- A **guided path** view: recommended study order across sections, each step = learn (links) → drill (practice set) → check (section quiz), with progress state.
- Homepage resume card: "Your progress: 5/8 sections touched · weakest: Estimator · continue →" (computed client-side from localStorage against precomputed thresholds).
- Acceptance: every objective in syllabus.json maps to ≥1 live resource + ≥1 practice set; all resource links liveness-checked in CI.

### F2 — Machine-verified question bank
- Target v1: **≥200 shipped questions** (goal 300), stratified by section weights, every one tagged: section, objective, difficulty (1-3), type, provenance.
- Types: multiple-choice, multi-select, predict-the-output, spot-the-bug, which-code-produces-X, scenario→best-practice.
- **Verification pipeline** (per question):
  1. Generated by an agent from objective + scope notes (+ docs links for grounding).
  2. Code-bearing: harness executes the stem code AND each option's claim against pinned Qiskit; correct answer must be *observed*, distractors must be *refuted* (different output or raised error as claimed). Proof artifact (code, observed output, versions, seed) stored with the question.
  3. Conceptual (no code): must carry ≥1 official doc citation; adversarial reviewer checks the claim against the cited page.
  4. **Adversarial review fan-out**: independent verifier agents try to kill the question (ambiguous stem? two defensible answers? version-sensitive? trivially guessable?). Only survivors ship. Kill-rate is recorded as a quality metric.
- Explanations: why-correct plus **per-distractor why-wrong**, with proof excerpt and doc links. This is the teaching surface.
- Schema: `data/questions/<section>/<qid>.json` (see §5). Freshness: CI re-executes the full bank on Qiskit bumps; failures → `STALE` flag, auto-excluded from site until regenerated.

### F3 — Practice player
- Per-section and per-objective drills; instant grading; explanation + proof reveal; flag-for-review.
- Mastery model per objective in localStorage (attempts, correct-streak, last-seen) → drives F1 progress and F5 recommendations.
- "Weakest first" smart drill = client-side selection from precomputed ordering tables (no runtime AI).

### F4 — Mock exam simulator
- Faithful format: **68 questions / 90 minutes / pass at 47**, sampled from the bank **stratified by official section weights**, seeded shuffle (shareable seed = reproducible exam), multi-select where the blueprint has them.
- Exam UX: timer, question palette, mark-for-review, no feedback until submit.
- Score report: per-section bars vs pass line, weakest-section diagnosis, and **precomputed remediation** (section × score-band → specific pages, drills, and resources to fix), attempt history + readiness trend chart (localStorage).
- Acceptance: sampling matches weights within ±1 question; timing and scoring exactly mirror the real exam's public parameters.

### F5 — Progress & readiness dashboard
- Objectives-covered map (visited/drilled/quizzed per objective), mock-exam trend, estimated readiness per section (precomputed thresholds; honest uncertainty wording).
- Exportable **study record** (markdown/JSON download) — useful for advocate applications and study groups.

### F6 — Community exam hub + validator
- Directory of all known advocate-created practice exams (from the advocate-library list + research inventory): link, author (GitHub identity), format, question count, license — with credit and thanks. We are an index, not a replacement.
- **Validator pipeline** (build-time): for each machine-readable exam (markdown/notebook), parse questions → execute code-bearing ones against pinned Qiskit 2.x → compare stated answer keys with observed behavior → emit a **respectful quality report** (per-exam page): parse coverage, #executed, #confirmed, #mismatches (with evidence), version sensitivities. Framing: "findings, PRs welcome", never a leaderboard of shame. Takedown-friendly: any author can request removal.
- Report doubles as **advocate outreach**: each finding is a ready-made contribution (issue/PR) for us or others to file upstream.
- v1 scope: parse+validate the markdown/notebook subset (≥5 exams), directory lists all.

### F7 — Flashcards + spaced repetition
- Build-time compiled decks per section: concept cloze, Q/A, predict-output cards (subset derived from the question bank + scope notes).
- In-site **Daily review** widget (SM-2 in localStorage, "N cards due" nudge on homepage).
- **Anki export**: `.apkg` per section + full deck, built with `genanki` in the pipeline, downloadable statically.

### F8 — Settings (incl. doQumentation integration, opt-in)
- Theme, display density, data export/import (full localStorage state as JSON), reset.
- **doQumentation integration — default OFF, explicit opt-in**: when enabled, (a) topic pages additionally deep-link to doqumentation.org learning material (incl. locale subdomains), and (b) *experimental*: in-browser code execution on practice pages via the same client-side stack doQumentation uses (thebelab → mybinder.org with QuBins pinned images — no doqumentation.org server involved; verified feasible for external origins). Clearly labeled experimental; cold-start caveat shown.

### F9 — Build/ops layer (the factory)
- **Drift watchdogs** (weekly CI): (1) re-execute the whole question bank against latest Qiskit 2.x → STALE queue; (2) link liveness for all resource URLs; (3) exam-page/study-guide change detection (hash + diff → issue).
- **Provenance & freshness badges** on site: "Bank verified against Qiskit 2.5.0 on 2026-07-19 · 0 stale".
- All generation/verification scripts in `pipeline/`, deterministic and re-runnable; agent-driven steps documented in `pipeline/README.md` so future Claude sessions can re-run them.

## 5. Architecture

```
/                      # repo root
  PRD.md               # this file
  data/
    syllabus.json      # canonical syllabus (build input #1)
    questions/         # verified question bank (one JSON per question)
    proofs/            # execution proof artifacts (referenced by questions)
    community-exams.json  # validator inventory + latest report data
    remediation.json   # precomputed section×band advice tables
  pipeline/            # Python: generation orchestration docs, verification harness, validator, anki build
  research/            # research reports (provenance)
  site/                # Docusaurus 3 (TypeScript)
  .github/workflows/   # build+deploy, weekly drift watchdog
```

**Question JSON schema (v1):**
```json
{
  "id": "s5-q014", "section": "s5", "objectives": ["s5o1"],
  "type": "mcq|multi|predict-output|spot-bug",
  "difficulty": 2,
  "stem": "…markdown…", "code": "…or null…",
  "options": [{"key":"A","text":"…"}],
  "answer": ["C"],
  "explanation": {"correct": "…", "distractors": {"A": "why wrong"}, "citations": ["url"]},
  "proof": {"status":"executed|conceptual","artifact":"proofs/s5-q014.json"},
  "provenance": {"generated":"2026-07-19","model":"claude","reviewed":false,"adversarial_rounds":2},
  "freshness": {"verified_against":{"qiskit":"2.5.0","runtime":"0.48.0"},"verified_on":"2026-07-19","stale":false}
}
```

**localStorage keys**: `qcs.mastery.<objective>`, `qcs.attempts.mock`, `qcs.srs.<deck>`, `qcs.settings`, `qcs.visited.<page>` — all export/importable.

**Verification harness**: `.venv` pinned (Qiskit 2.5.0 / Aer 0.17.2 / Runtime 0.48.0 at creation; exact versions recorded per run). Fake backends (`FakeBrisbane` etc. from `qiskit_ibm_runtime.fake_provider`) replace real hardware; seeds fixed; shots clamped — pattern adopted from doQumentation's `00_patch_runtime.py` CI harness. Runner API: give it a question JSON → it returns proof/refutation per option.

## 6. Quality gates (build fails otherwise)

1. Bank invariants: unique IDs, schema-valid, every section ≥ its weight-proportional minimum, no question without explanation+citation/proof.
2. 100% of code-bearing questions executed successfully; 0 shipped questions with failed/missing proofs; 0 STALE questions in a release build.
3. Mock-exam sampler unit tests (stratification, seeding, scoring incl. multi-select rules).
4. Link liveness ≥ warn-threshold; broken official links block release.
5. Site build with `onBrokenLinks: 'throw'`; a11y smoke checks on player components.
